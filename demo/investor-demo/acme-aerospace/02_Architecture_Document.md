# AASCRS Architecture Document

**System name:** Acme Aerospace Supply Chain Risk System (AASCRS)
**Architecture version:** 0.7 draft
**Author:** Daniel Park, Lead Architect
**Reviewers:** Maria Chen (ISSO), Akshay Bhattacharya (CTO)

---

## 1. Architecture Overview

AASCRS is a three-tier web application:

- **Presentation tier** — a React single-page application served from
  CloudFront, talking to the API tier via signed HTTPS calls
- **Application tier** — a FastAPI service running on AWS Elastic Container
  Service (ECS) with Fargate, sized at 4 vCPU / 16 GB RAM per task
- **Data tier** — Amazon RDS for PostgreSQL 16 (Multi-AZ, automated backup
  retention 35 days) and an S3 bucket for ingested supplier filings

All three tiers run inside a dedicated VPC (`vpc-aascrs-prod`) in the
`us-gov-west-1` region with no public subnets except for the load-balancer
subnet, which is restricted by security group to the AWS WAF and the Acme
corporate IP range.

---

## 2. Authentication Architecture

Authentication is performed by the Acme corporate identity provider (Okta
Workforce Identity Cloud, GovCloud edition). The AASCRS application is a
SAML-2.0 service-provider in Okta.

The Acme Okta tenant requires **PIV smart-card authentication for all
federal personnel** and for any account designated as `privileged`. Non-
federal contractor accounts (Acme employees) authenticate with username +
password followed by a TOTP code generated by Okta Verify. Hardware
security keys (FIDO2) are accepted as a fallback.

Session length is fixed at 8 hours. Refresh tokens are not issued; users
must re-authenticate after the session ends.

The Information System Security Officer (ISSO) role is delegated through an
Okta group. Group membership is reviewed every 90 days by the ISSO and the
procurement-system-owner jointly.

Egress from the application tier to the public internet is denied by VPC
routing; the only outbound flows are to AWS service VPC endpoints (S3,
KMS, Secrets Manager) and a single allow-listed flow to the supplier-data
ingest API.

---

## 4. Boundary Protection

The AASCRS authorization boundary terminates at:

- **Inbound:** The AWS Application Load Balancer in the load-balancer
  subnet. The ALB enforces IP allow-listing to Acme corporate IP ranges
  and rejects any other source IP at the security-group layer.
- **Outbound:** VPC endpoints for AWS services and the supplier-data API
  egress. No general internet access from the application tier.

All traffic crossing the boundary is logged to AWS VPC Flow Logs and
retained for 365 days in the AASCRS audit S3 bucket.

---

## 5. Data Encryption

| Layer | At-rest | In-transit |
|---|---|---|
| RDS PostgreSQL | KMS-managed AES-256 | TLS 1.3 from app tier |
| S3 supplier filings | KMS-managed AES-256 | TLS 1.3 |
| Secrets Manager (DB creds, API keys) | KMS-managed AES-256 | TLS 1.3 |
| ALB to client | n/a | TLS 1.3 (ACM cert) |

The KMS key (`alias/aascrs-prod`) is owned by the AASCRS account and key
rotation is enabled (annual automatic rotation).

---

## 6. Monitoring & Observability

- **Application logs** stream to CloudWatch Logs and are forwarded to the
  Acme SOC's Splunk instance for SIEM correlation.
- **Network flow logs** are captured at the VPC level (above) and
  ingested into Splunk for analysis.
- **AWS GuardDuty** is enabled at the account level and forwards findings
  to the SOC.
- **AWS CloudTrail** is enabled and writes to a separate audit account.

Alerts are forwarded by the SOC to the AASCRS on-call analyst with a
target acknowledgement time of 30 minutes.

---

## 7. Inheritance & Shared Responsibility

AASCRS inherits the following controls from the AWS GovCloud platform under
the AWS FedRAMP Moderate authorization (current as of FedRAMP package
2024-03):

- **PE-3** Physical Access Control (data-center physical access)
- **PE-6** Monitoring Physical Access
- **MA-2** Controlled Maintenance (AWS hardware maintenance)
- **CM-4** Impact Analyses (AWS platform-level changes)

These are documented as inherited in the AASCRS POA&M template and a copy
of the FedRAMP authorization letter is on file with the AO.

---

## 8. Out of Scope

This architecture document does **not** describe:

- The Acme corporate identity provider topology (Okta tenant configuration)
- The SOC's internal Splunk architecture
- The supplier-data ingest API's own architecture

These are referenced as upstream / downstream dependencies.
